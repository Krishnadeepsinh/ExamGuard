6–8 hrs

90M+

Stylometric baseline — 3-tier strategy for cold-start (new · fixes Gap 2)
First exam, second exam, veteran student. Each handled differently. No silent failures.
The single most dangerous gap in v5. A judge who asks "what happens for a student's first exam?" and gets no answer will mentally fail the submission. v6 has a real answer for every case.
Tier 1 · Full Detection
Condition: baseline_answer_count ≥ 3 prior exams
Full stylometric detection active. Style baseline built from average of all prior exam answers. style_distance computed normally. Stylometric factor = 25% weight in integrity score. Confidence interval: ±7. This is the standard operating mode for returning students.
Tier 2 · Partial Detection
Condition: baseline_answer_count = 1–2 prior exams
Stylometric detection active but downweighted. Style baseline from limited prior answers. Wider confidence interval (±15 instead of ±7). Stylometric factor weight reduced from 25% to 10%. Remaining 15% redistributed to behavioral (30%→40%) and answer quality (25%→30%). Teacher sees: "Stylometric: limited baseline — detection less precise." No false positives from small sample.
Tier 3 · Disabled (First Exam)
Condition: baseline_answer_count = 0 (first exam on platform)
Stylometric factor completely disabled. Integrity score recalculated as 4-factor only: behavioral (37.5%), AI-perplexity (18.75%), answer quality (31.25%), time-anomaly (12.5%). Teacher sees: "⚠ First exam — stylometric detection unavailable. Score based on 4 signals." This exam's answers contribute to building the student's baseline for future exams. Honest, not a bug.
DB change — baseline_answer_count column
Add baseline_answer_count INT DEFAULT 0 to the users table. After each completed exam session, increment this counter. The Stylometric Agent reads this value at session start to determine which tier applies. This is one migration, one column, one read. No performance impact.
Integrity report — show tier visibly
Every integrity report PDF must show the baseline tier: "Stylometric detection: Tier 1 (full baseline, 3 prior exams)". This tells the teacher exactly how much to trust the stylometric component. Transparent, auditable, and honest. Judges will respect this far more than a system that claims full detection on exam #1.

Orchestrator state machine — deterministic, fully auditable
No LLM decides when to flag a student. Every threshold is a number.
The Orchestrator never uses an LLM to decide escalation. It is a pure rule-based state machine. The LLM is invoked only to generate the human-readable explanation text shown to the teacher. This is your answer to "how do you prevent AI bias in your flagging system?"
WATCH
Elevated attention. Log but don't interrupt.
integrity_score ≤ 85 OR
behavioral_events ≥ 1 (gaze >4s, tab switch, audio spike) OR
time_anomaly: easy Q <15s OR
first tab-switch detected
Action: Log event to Redis. Update tile on teacher dashboard (blue border). No student notification. Exam continues normally.
WARN
Significant signal. On-screen warning to student.
integrity_score ≤ 70 OR
behavioral_events ≥ 2 in 10-min window OR
perplexity_score ≤ 0.45 (one answer) OR
style_distance ≥ 0.50 (one answer, tier 1/2 only) OR
medium Q answered in <45s OR
face not detected >15s
Action: Show student "Academic integrity warning. Continue honestly." Teacher tile turns amber. warn_count++. Alert feed entry logged.
FLAGGED
High-confidence signal. Human review required.
integrity_score ≤ 50 OR
(perplexity ≤ 0.30 AND style_distance ≥ 0.65) OR
behavioral_events ≥ 4 total OR
warn_count ≥ 3 OR
phone_detected on webcam OR
hard Q (≥8 marks) answered in <60s AND perplexity ≤ 0.5
Action: session.status = 'flagged'. Grade held. Review Agent activated. Teacher notified. AI never auto-fails — teacher is always final decision-maker.
Critical design principle — AI flags, teacher decides, student can appeal
The Orchestrator never disqualifies a student. It sets session.status = 'flagged' and holds the grade. Teacher reviews the integrity report. Student sees an anonymized summary and can submit a 200-word appeal. Teacher sees both side-by-side and presses "Confirm Flag" or "Clear." This answers the false positive question completely. The 8% FPR is acceptable because no automated adverse decision is ever final — DPDP Article 8 compliant.

Grade appeal flow — new v6 · handled by Review Agent
Student sees flags. Student can respond. Teacher decides with full context.
Missing from v5. In a real deployment, a student whose grade is held has rights. This flow closes the ethical loop and gives judges the answer to "what does the student see when they're flagged?"
Student-facing appeal flow
01
Exam ends. Student sees: "Your exam has been submitted. Your teacher is reviewing your results." No score shown yet. No accusation language.
02
Review Agent sends anonymized integrity summary to student: "Your session raised 2 integrity signals: (1) Answer to Q3 showed unusual writing patterns. (2) Tab switch detected at 14:32." Raw biometric data never shown to student.
03
Student can optionally submit a 200-word written response explaining their side. No obligation. Deadline: 24 hours. After deadline, teacher can proceed without it.
04
Teacher receives notification: "Student [name] has submitted a response to the integrity review. View side-by-side." Teacher sees integrity report + student response together.
05
Teacher presses "Confirm Flag" or "Clear Flag." Review Agent triggers grade release via Supabase RPC. Student sees final result. Decision logged permanently with teacher ID and timestamp.
What the student summary does NOT include
Raw gaze vectors, exact EAR readings, perplexity scores, style_distance values, or any raw biometric data. Students see plain-language descriptions of flags only: "unusual writing patterns," "tab switch detected," "answer submitted very quickly for a 10-mark question." This protects against gaming while being fair and transparent.

Judges at international hackathons increasingly look for teams that think about edge cases and ethics, not just the happy path. "What does the falsely flagged student experience?" is a question many teams cannot answer. You can answer it in full. The appeal flow also makes your 8% FPR acceptable — because every false positive is catchable before it becomes a real harm.

Bloom's taxonomy prompt templates — new v6 · fixes Gap 8
Six real prompts. One per Bloom level. Question Agent injects them at generation time.
v5 stored bloom_level in the paper_config but never used it to change the generation prompt. v6 fixes this with a concrete template library. These are the exact prompt fragments the Question Agent prepends when generating for each Bloom level. ~30 lines of code. Real differentiation.
Level 1
Remember
Write a question that asks students to recall, name, define, or list [concept] exactly as it appears in the provided text. The answer must be directly stated in the source material.
Level 2
Understand
Write a question that asks students to explain, summarise, or describe [concept] in their own words, demonstrating they understand what the source material is saying, not just what it says.
Level 3
Apply
Write a question that gives students a new scenario or numerical problem and asks them to use [concept] from the source material to solve it. The scenario must not appear in the text but requires the concept to answer.
Level 4
Analyse
Write a question that asks students to break down [concept] into its components, identify relationships or patterns, distinguish between two similar ideas, or find the cause of an outcome described in the source material.
Level 5
Evaluate
Write a question that asks students to make a judgement about [concept], assess the validity of a claim using evidence from the source material, or compare two approaches and argue which is better with justification.
Level 6
Create
Write a question that asks students to design, propose, construct, or produce something new using [concept] from the source material. The student must synthesise ideas to produce an original response, not recall an existing one.
How the Question Agent uses this at runtime
The Question Agent has a BLOOM_TEMPLATES dict keyed by level. At generation time: template = BLOOM_TEMPLATES[section.bloom_level]. The template is prepended to the RAG-context prompt: f"{template}\n\nSource material:\n{retrieved_chunks}\n\nGenerate a {section.qtype} question worth {section.marks_each} marks." This ensures "remember" and "create" level questions are genuinely different in cognitive demand — not just labeled differently. The difference is visible, demoed live, and real.

Judge FAQ — v6 (cold-start + consent + agent count added)
Every likely probe. Every answer cold-ready.
"How is this different from Proctorio or Mettl?"
Five things they can't do: (1) Paper Configuration Engine — no existing tool lets a teacher configure marks per section, question type, Bloom's level, and chapter weightage in one UI. (2) RAG-grounded generation from the teacher's own material — questions trace back to a page in the teacher's PDF. (3) AI-answer detection via stylometrics + perplexity — they watch faces, not text. (4) Local LLM for biometrics — DPDP compliant, no US cloud. (5) Flat ₹2,000/month vs ₹800/student/test. We solve problems Proctorio hasn't even defined yet.
"What happens for a student on their very first exam?"
We handle this explicitly with a 3-tier baseline strategy. Tier 3 (first exam): stylometric detection is completely disabled — the student has no prior baseline, so running detection would produce meaningless style_distance values. The integrity score is recalculated as a 4-factor score, the teacher sees "First exam — stylometric unavailable" on the dashboard tile, and this exam's answers contribute to building the student's baseline for future exams. No false positives from missing data, no silent failures, and the system is honest about what it can and cannot detect.
"What's your false positive rate? You'll fail honest students."
FPR = 8% on our N=60 calibration dataset, documented in research/calibration/eval_notebook.ipynb. We know this. That's exactly why the system never auto-fails anyone. The Orchestrator sets status = 'flagged' and holds the grade — the teacher is always the final decision-maker. Additionally, the flagged student receives an anonymised summary and has 24 hours to submit a written appeal. Teacher sees both side-by-side. At 8% FPR with 100 students, ~8 answers get reviewed by the teacher — far less work than manually invigilating 100 students, and every decision has a human behind it.
"You claim 10 agents — what does each one actually do?"
Each agent has a single, specific job: Paper Config Agent validates marks budgets and chapter coverage. Material Ingestion Agent parses PDFs including OCR. Orchestrator routes events deterministically (no LLM). Question Agent generates typed questions from RAG chunks with Bloom's templates. Proctoring Agent processes webcam data in browser WASM. Stylometric Agent applies 3-tier baseline detection. Security Agent monitors tab switches, paste, audio, time anomalies. Evaluation Agent grades and evaluates against rubric. Report Agent generates PDF reports. Review Agent manages appeals and grade release. Open LangGraph Studio right now — you can watch every agent fire.
"What if the student doesn't consent to the monitoring?"
They cannot enter the exam. The consent screen is a gate — session.consent_given must be true before the Proctoring Agent starts. If the student cancels, no session is created. Teachers can configure "no-webcam mode" for students with documented accessibility reasons — the teacher manually approves this, and the integrity score is calculated without behavioral signals, clearly marked. Consent is logged with a timestamp for audit.
"What if the internet drops mid-exam?"
Every answer is saved to localStorage every 30 seconds. On reconnect, the client diffs local state with server and resyncs silently. If disconnection exceeds 5 minutes, the session is paused — not terminated — and the teacher is notified. Student sees "Your session has been paused. Contact your teacher." Teacher can resume manually. We don't punish students for Indian network infrastructure we don't control.

After each exam, analyse class performance per chapter and section type. Suggest adjusted difficulty distribution for the next paper. "Section D average was 92% — increase to Hard next time." Closes the pedagogical feedback loop.
Generation queue + progress bar +
For large papers (80+ questions), generation currently runs synchronously. A background job queue (Celery + Redis) with a teacher-facing progress bar ("Generating question 23/80... est. 4 min remaining") prevents timeouts on Gemini rate limits and gives teachers confidence the system is working.
Stylometric Tier 2 calibration (v7) +
The current Tier 2 (1–2 prior exams) uses a manually specified wider CI (±15). A calibrated transition function that narrows the CI as baseline_answer_count grows from 1 to 3 would make the detection more precise for borderline cases without adding false positives. Post-hackathon research item.
