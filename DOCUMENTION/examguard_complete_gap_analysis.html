Critical UX patterns not in your spec
1. Error states — completely unspecced Critical
Every screen needs an error state. Your spec has none. Judges will test failure paths.
Add these to each screen:
• Screen 3 (Paper Config): "Question generation failed — Gemini rate-limited. Switching to Ollama..." with spinner and ETA
• Screen 3: "OCR failed on page 4. Skipping and continuing with 280/340 chunks." with page numbers listed
• Screen 4 (Live Monitor): "WebSocket disconnected. Reconnecting... (attempt 2/5)" banner at top
• Screen 7 (Student Exam): "Connection lost. Your answers are saved locally. Reconnecting..." with last-save timestamp
• Screen 10 (Reports): "PDF generation failed. Retry" per-student button
2. Empty states — not designed Critical
New teachers see a completely blank dashboard. No spec for this first-run experience.
Design these empty states:
• Teacher Dashboard (no exams): Illustration + "Create your first exam" CTA with a 3-step guide inline
• Live Monitor (exam not started / no students joined): "Waiting for students... Share code: ABC123" with copy button
• Reports tab (exam in progress): "Reports available after exam ends" with live countdown
• Paper Config (no material uploaded): "Upload your syllabus first" with upload zone visible inline
3. Mobile layout — critical gap for Indian students Critical
Students in India use phones. Screen 7 (Exam Interface) has no mobile spec. Camera permission flow on mobile is different (Safari iOS vs Chrome Android).
Mobile-specific requirements to add:
• Screen 7 (Exam): Single-column layout, question at top, answer below, large tap targets for MCQ options (min 48px)
• Screen 6 (Blink): Camera preview fullscreen on mobile with large instruction text
• Screen 5 (Consent): Scrollable modal — 4 consent items may overflow on small screens
• Screen 4 (Live Monitor): Responsive tile grid — 1 column on mobile, 2 on tablet, 4 on desktop
• MediaPipe WASM: Mobile Safari blocks camera until user gesture — handle permission request before consent screen dismissal
4. Loading & skeleton states Important
Plan mentions a progress bar during question generation, but no loading states for anything else.
Add skeleton loaders to:
• Teacher Dashboard: Skeleton exam cards while exams API loads
• Paper Config preview: Skeleton question cards while generation runs
• Reports page: Skeleton student cards while reports API loads
• Live Monitor tiles: Skeleton tiles for the first 500ms before WebSocket connects
5. Toast / notification system Important
No notification system is specced. Many events need user feedback.
Events that need toasts (use react-hot-toast or shadcn/ui Sonner):
• ✓ Success: "Join code copied", "Exam activated", "PDF report ready", "Appeal submitted"
• ⚠ Warning: "LLM fallback activated — using Ollama", "Low chunk count for Chapter 5"
• ✗ Error: "Upload failed — file too large", "Session expired — please log in again"
• ℹ Info: "WebSocket reconnected", "3 students are now waiting to join"
6. Question navigation panel (Screen 7) Critical for UX score
Previous/Next only is unacceptable for a 40-80 question exam. Every Indian student expects a question palette.
Add to Screen 7:
• Side panel (or bottom drawer on mobile): numbered grid of all questions
• Color coding: white = unvisited, blue = answered, yellow = marked for review, red = flagged by integrity system
• Summary row: "Answered: 22/40 | Marked: 4 | Unanswered: 14"
• "Submit All" confirmation dialog showing summary before final submit
7. Live Monitor UX improvements Judges will see this
Current spec is functional but not polished enough for a judging demo.
Improvements to add:
• Sort tiles by: score (ascending — show most at-risk first), name, join time
• Filter tiles by: WATCH / WARN / FLAGGED / All
• Summary bar above tiles: "32 active | 2 WARN | 1 FLAGGED | Avg integrity: 81"
• Sound alert toggle for FLAGGED events (teacher may not be watching screen)
• "End Exam for All" button with confirmation — not in spec but needed for teacher control
8. Integrity score display consistency Confusing without this
Score shown in multiple places (tile, expanded view, PDF report, appeal panel) but the visual format isn't consistent across your spec.
Define one canonical score display component:
• Score number (large, color-coded) + CI range (e.g. "74 ±7") + tier badge ("Tier 1")
• 5 factor bars with labels and weights (Behavioral 30%, Stylometric 25%, etc.)
• Status badge: CLEAN / WATCH / WARN / FLAGGED
• Use this same component in: tile expand, PDF report, appeal panel, student post-exam screen
9. Paper Config UX — marks tally needs work Demo moment
The live marks tally is a great feature but the current spec doesn't describe the micro-interaction clearly enough to implement well.
Specify the tally micro-interaction:
• Tally updates on every keystroke in count/marks fields (not on blur)
• Show section breakdown: "Section A: 20 | Section B: 30 | Section C: 15 → Total: 65/80"
• Over-budget: tally turns red + "5 marks over — reduce a section" error inline
• Under-budget: tally turns amber + "15 marks remaining — add a section or increase marks"
• Exactly right: tally turns green + checkmark animation + "Generate Paper" button activates
10. Design tokens — partially defined Consistency issue
§10 mentions design tokens but only lists color names, not the full token system needed for consistent implementation.
Add these missing token definitions:
• Typography scale: display (32px/700), heading (24px/600), subheading (18px/500), body (15px/400), caption (13px/400), mono (13px/400)
• Spacing scale: 4px, 8px, 12px, 16px, 24px, 32px, 48px, 64px — use consistently
• Border radius tokens: sm=4px, md=8px, lg=12px, xl=16px, pill=9999px
• Shadow tokens: card (0 1px 3px rgba), modal (0 8px 32px rgba), elevated (0 4px 12px rgba)
• Animation tokens: fast=150ms, normal=250ms, slow=400ms — use for all transitions
11. Accessibility — not mentioned anywhere Judges with a11y knowledge will notice
Minimum accessibility spec to add:
• All live integrity updates use aria-live="polite" so screen readers announce changes
• Timer countdown announced at 5 min, 1 min, 30 sec remaining via aria-live="assertive"
• Focus trap on consent modal — tab key must not escape the modal
• All icon-only buttons have aria-label (copy button, alert dismiss, tile expand)
• Color is never the only integrity status indicator — pair with icon (✓ ⚠ ✗) and text label
12. Onboarding / first-run experience Judges sign up fresh
Judges will create a new account during evaluation. They will land on an empty dashboard with no guidance.
Add a first-run flow:
• After signup: 3-step checklist shown inline: "1. Create your first exam → 2. Upload your syllabus → 3. Configure your paper"
• Pre-loaded demo: seed_demo.py already creates a ready-to-demo environment — show the judge this option at login: "Try demo mode?" with one click
• Progress indicator: show which steps are completed with checkmarks as judge explores

Infrastructure gaps
Missing infra specs Before submission
Rate limiting not specced — The POST /sessions/join endpoint is public (no auth needed). Any student with a valid join code can join. No rate limiting = trivial DDoS on the demo server. Add: slowapi (Python rate limiter) on join endpoint, max 5 attempts per IP per minute.
Redis TTL not specced — Session state in Upstash Redis needs an expiry. A 60-minute exam session's Redis keys should expire after 90 minutes. Not in spec.
Supabase Storage bucket policy not specced — PDF reports are uploaded to Supabase Storage. The bucket access policy (public vs private + signed URLs expiry duration) is not defined.
CORS policy not documented — FastAPI CORS settings for Vercel domain need to be explicit. A common demo-day failure is Render backend rejecting requests from the Vercel frontend URL.
WebSocket reconnection strategy not specced — Socket.IO client should have explicit reconnection config: max attempts, exponential backoff, and what state to rehydrate after reconnect.
