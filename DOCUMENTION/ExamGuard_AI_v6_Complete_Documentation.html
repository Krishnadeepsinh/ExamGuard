§ 0 · Executive Summary
ExamGuard AI — Project Overview
A comprehensive overview of the product vision, problem statement, solution, and high-level metrics for ExamGuard AI v6.
Problem Statement
Three critical failures in India's exam ecosystem
⏱ 8-Hour Paper Creation
Teachers spend 6–10 hours manually typing and formatting exam papers, searching question banks, and ensuring curriculum alignment. No intelligent automation exists for Indian schools.
🤖 AI Cheating Is Invisible
Students can submit GPT-generated answers with zero detection. Existing proctoring tools check only behavioral signals — they cannot detect AI-written text at all. FPR of legacy tools exceeds 30%.
💸 Proctorio Costs Are Prohibitive
Enterprise proctoring solutions like Proctorio cost ₹2,000–5,000 per student per exam. For India's 90 million exam-takers, this is economically out of reach for 95% of institutions.
Solution
ExamGuard AI — One platform. Three superpowers.
What ExamGuard AI Does
1. Automated Paper Generation — Teachers upload their own textbook PDFs. A 10-agent LangGraph pipeline ingests, chunks, embeds, and generates a fully customized exam paper with configurable sections, question types, marks per question, Bloom's taxonomy levels, and chapter weightage — all from the teacher's actual material, not generic question banks.

2. Multi-Signal AI Cheat Detection — A 5-factor integrity scoring engine combining behavioral proctoring (gaze, tab, audio), perplexity/burstiness scoring for AI-written text, stylometric baseline comparison, answer quality evaluation, and time anomaly detection. 91% TPR, 8% FPR, validated on N=60 dataset.

3. Fair & Transparent Review Flow — Flagged students see an anonymized integrity summary and have a 24-hour appeal window. Teachers receive both sides and make the final decision. Grade appeals are tracked and auditable. Built on DPDP-compliant consent architecture.

Module 2 · Material Ingestion (RAG Pipeline)

FR-MAT: Material Upload & Embedding

1
File Upload & Validation
Accepts PDF, DOCX, TXT. Max size: 50MB. Validates MIME type and file header. Extracts text via PyMuPDF for text-based PDFs. Detects if PDF is scanned (text layer <50 chars/page) and routes to OCR path.
2
OCR Processing (Scanned PDFs)
pytesseract (local, primary) or Google Vision API (if GOOGLE_API key set, fallback). If both fail, teacher is prompted to paste text manually. OCR output undergoes post-processing to remove artifact characters. Processing time displayed to teacher in real-time.
3
Chapter Detection & Chunking
Regex + heuristic heading detector identifies chapter boundaries. Text chunked at 512 tokens with 64-token overlap. Each chunk tagged with: chapter_id, page_start, page_end, chunk_index. Teacher sees chunk count and chapter map preview before confirming.
4
Embedding & Vector Storage
sentence-transformers (all-MiniLM-L6-v2) generates 384-dim embeddings locally. Stored in Supabase pgvector extension. Similarity search uses cosine distance (ivfflat index). Embedding time: ~2s per page on CPU. Full 50-page NCERT chapter: <90 seconds total pipeline.

Module 4 · Proctoring System

FR-PROC: Real-time Behavioral Monitoring

Signal 1 · Gaze Tracking
Tool: MediaPipe FaceMesh (WASM, runs in browser)
Metric: Eye Aspect Ratio (EAR) for blink + gaze vector for off-screen look
Threshold: Gaze away >4 consecutive seconds = 1 gaze event
Escalation: 3+ gaze events in exam → behavioral score drops 15pts per event
Signal 2 · Tab Switch Detection
Tool: document.visibilityState + focus/blur events (browser JS)
Metric: Count of tab-away events, duration per absence
Threshold: Any tab switch = 1 event. >3 events → WARN trigger
Note: Students informed of this in consent screen. No silent detection.
Signal 3 · Audio Level
Tool: Web Audio API (browser, microphone access)
Metric: RMS amplitude level; whisper spikes above threshold
Threshold: Audio level >0.45 RMS for >2 seconds = 1 audio spike
Privacy: Audio NOT recorded. Level only. Processed locally in browser.
Privacy Architecture — All biometric processing is local-only
MediaPipe WASM runs entirely in the student's browser. No webcam frames are sent to the server. No audio is recorded. Only numerical signals (gaze vector coordinates, EAR values, audio RMS levels) are transmitted as structured events. This is documented in the DPDP compliance checklist and shown to students on the consent screen. No exceptions — biometric data never leaves the student's device.
Module 5 · Integrity Scoring Engine (Deterministic)

FR-INT: 5-Factor Score Calculation

INTEGRITY ENGINE SPECIFICATION
5-Factor Weighted Score · Deterministic Thresholds · No LLM for Escalation Decisions
54
Example Score · FLAGGED State
Factor 1: Behavioral (weight: 30%) — Baker 2019 methodology
65
Score = 100 − (gaze_events×15) − (tab_events×10) − (audio_spikes×8). Floor at 0. 2× gaze away, 1× tab, 1× audio spike → 65.
Factor 2: AI-Cheat Perplexity (weight: 15%) — Perkins 2023 methodology
28
Ollama logprobs → perplexity score per answer. Low perplexity (AI text) = low score. Burstiness metric for sentence-length variance. Q3: perplexity 18.4, burstiness 0.19. Fallback: Shannon entropy if logprobs unavailable.
Factor 3: Stylometric Baseline (weight: 25% Tier1 / 10% Tier2 / 0% Tier3)
38
sentence-transformers cosine similarity vs student's baseline. Distance threshold: >0.60 = outlier. Q3 style_dist 0.73 vs baseline 0.11. Tier 1 CI: ±7. Tier 2 CI: ±15.
Factor 4: Answer Quality — Gemini rubric evaluation
85
Gemini 1.5 Flash evaluates factual accuracy, coherence, and rubric alignment (0–100). High quality alone does not trigger flag. Used in context with other factors.
Factor 5: Time Anomaly (weight: 5%) — Added in v5
20
If marks ≥ 5 AND time_spent_seconds < 60 → auto-WARN trigger. Q3 (10m essay) answered in 42 seconds. Score = max(0, 100 − (expected_min_time/actual_time)×100).
StateScore RangeColorActionGrade
WATCH80–100GreenNo action. Normal monitoring continues.Released immediately after exam
WARN60–79AmberTeacher notified. Teacher may choose to review manually.Teacher can release or flag for review
FLAGGED0–59RedGrade held. Student receives anonymized summary + appeal window (24hr). Teacher must confirm or clear.Held until teacher decision
Critical Design Decision — LLM Never Makes Escalation Decisions
The WATCH/WARN/FLAGGED thresholds are deterministic and numeric. No LLM decides whether to flag a student. LLMs are used only for: (1) generating human-readable explanation text in the report, (2) evaluating answer quality rubric scores. The escalation decision is always rule-based and auditable. This is a deliberate design choice to prevent AI hallucinations from creating unfair outcomes for students.
Module 6 · Grade Appeal Flow

FR-APPEAL: Student Appeal & Teacher Review

F
Trigger: Student FLAGGED
Integrity score <60. Grade held. review_status = 'pending'. System generates anonymized integrity PDF (student version — no raw biometric data, plain-language flags only).
N
Student Notification
Student receives anonymized summary in their student portal. Shows: which signals triggered, Bloom's level of flagged questions, 24-hour countdown timer to submit appeal. Student submits written response (max 500 words) via integrity_appeals table.
R
Teacher Review Interface
Teacher sees: full integrity report (with raw data) on left panel, student appeal response on right. Teacher makes decision: CONFIRMED (flag stands, grade deducted per institution policy) or CLEARED (flag removed, original grade released). Decision logged with timestamp.
G
Grade Release
teacher_decision set to 'confirmed' or 'cleared'. decision_at timestamp logged. Grade released to student. review_status = 'resolved'. If no teacher decision within 48 hours, system sends escalation reminder. Appeal records retained for 6 months per DPDP compliance.

Security Requirements
🔐 Authentication Security
JWT tokens with 1-hour expiry. Refresh token rotation. Supabase Auth handles credential storage (bcrypt hashed, never plaintext). RLS prevents cross-tenant data access at DB level. All API endpoints require valid JWT except /health.
🛡 DPDP Compliance
India's Digital Personal Data Protection Act 2023. Explicit consent before data collection. Data minimization: audio level only (no recording). Retention: appeal records 6 months, exam sessions 1 year. Right to erasure: student data deleted on request. docs/dpdp-compliance.md in repo.
🔒 Data Isolation
Row-Level Security on all Supabase tables. Teachers cannot access other teachers' exams. Students cannot access other sessions. Biometric processing in browser only — never transmitted. Uploaded PDFs stored with signed URLs (1-hour expiry for access).
🔑 API Security
CLOUD_API_KEY for Gemini stored as environment variable only. Never in code or repository. .env.example shows required variables without values. GitHub Actions secrets for CI deployment. No API keys in commit history.
🌐 Transport Security
HTTPS enforced on all endpoints. WebSocket connection over WSS. Supabase connection via SSL. CORS configured: only frontend domain and localhost:5173 allowed. HSTS header on production deployment.
📋 Audit Trail
All integrity events stored with timestamps. All teacher decisions (confirm/clear) logged with decision_at. Commit history shows build progression (judges can verify). LangGraph trace available in LangGraph Studio screen recording.

Perplexity scoring for AI cheat detection. Returns per-token log probabilities. Low perplexity = AI-generated text. Fallback: Shannon entropy on answer text if logprobs unavailable (model-dependent).

Data / Storage

Bloom's Taxonomy Prompt Templates
6 concrete prompts — one per level — injected by Question Agent at runtime
Level 1
Remember
Write a question that asks students to recall, name, define, or list [concept] exactly as it appears in the provided text. The answer must be directly stated in the source material.
Level 2
Understand
Write a question that asks students to explain, summarise, or describe [concept] in their own words, demonstrating they understand what the source material says.
Level 3
Apply
Write a question that gives a new scenario or numerical problem and asks students to use [concept] to solve it. Scenario must not appear in text but requires the concept.
Level 4
Analyse
Write a question that asks students to break down [concept], identify relationships or patterns, distinguish between two similar ideas, or find the cause of an outcome in the source material.
Level 5
Evaluate
Write a question that asks students to make a judgement about [concept], assess the validity of a claim using evidence, or compare two approaches and argue which is better with justification.
Level 6
Create
Write a question that asks students to design, propose, construct, or produce something new using [concept]. Student must synthesise ideas to produce an original response, not recall an existing one.

3-Tier Stylometric Baseline Strategy
Tier 1 · Full Detection
baseline_answer_count ≥ 3 prior exams
Full stylometric detection active. Style baseline built from average of all prior exam answers. style_distance computed with sentence-transformers cosine similarity. Stylometric factor = 25% weight. Confidence interval: ±7. This is the standard operating mode for returning students. Report shows: "Stylometric: Tier 1 (full baseline, N prior exams)."
Tier 2 · Partial Detection
baseline_answer_count = 1–2 prior exams
Stylometric detection active but downweighted. Baseline built from limited prior answers. Wider confidence interval (±15 instead of ±7). Stylometric factor weight reduced: 25% → 10%. Remaining 15% redistributed: behavioral 30%→40%, answer quality 25%→30%. Teacher sees: "Stylometric: limited baseline — detection less precise."
Tier 3 · Disabled (First Exam)
baseline_answer_count = 0 (first exam on platform)
Stylometric factor completely disabled. Score recalculated as 4-factor only: behavioral (37.5%), AI-perplexity (18.75%), answer quality (31.25%), time-anomaly (12.5%). Teacher sees: "⚠ First exam — stylometric detection unavailable. Score based on 4 signals." This exam's answers build the student's baseline for future exams.

§ 12 · UI/UX Design System
Design Language, Component Library & Screen Specifications
Complete design system for ExamGuard AI. Covers design principles, color system, typography, spacing, components, and detailed screen specifications for all major views.
Design Principles
🔬 Clarity Over Cleverness
Teachers and students need to understand what the system is doing at all times. Every alert, badge, and score must be instantly interpretable. No ambiguous icons. Text labels everywhere. Error messages in plain English (or Hindi).
🤝 Trust Through Transparency
Students are being monitored. They must feel it's fair, not adversarial. Every monitored signal is disclosed upfront. Every flag comes with an explanation. Every decision has an appeal path. The system design should communicate: "We treat you like an adult."
⚡ Low Cognitive Load
Students are stressed during exams. Teachers are busy. The UI must reduce decision fatigue. Default choices should be correct 90% of the time. Progressive disclosure — show advanced options only when needed.
📱 Mobile-First, Keyboard-Second
Many Indian students access exams on smartphones. All student-facing UI must work on 360px minimum viewport. Tap targets ≥44px. No hover-only interactions. Teacher UI can be desktop-first with mobile fallback.
♿ Accessible by Default
WCAG 2.1 AA minimum. All interactive elements keyboard navigable. Color is never the only signal (badges have text labels). Screen reader compatible aria-labels on all form elements. Minimum 4.5:1 contrast ratio for body text.
🔄 Resilient to Network Failure
Indian internet is unreliable. Every action that writes data must show immediate optimistic UI update + confirmation when server responds. Network error states must be friendly, not terrifying. Reconnect flow must be seamless and non-destructive.

Component Specifications
Integrity Score Badge — State-aware color system
Props: score: number, state: 'WATCH' | 'WARN' | 'FLAGGED', ci: number, tier: 1|2|3
Variants:
  WATCH (80–100): Green background (#ecf7f2), border (#7ecaa0), score in dark green (#0f6e38)
  WARN (60–79): Amber background (#fdf8e6), border (#e8cc70), score in amber (#9a6f00)
  FLAGGED (<60): Red background (#fdf1f0), border (#f0a09a), score in red (#c0291b), pulsing border animation
Always shows: score/100, state badge, ±CI, baseline tier indicator
Accessibility: aria-label="Integrity score {score} out of 100, state {state}"
Bloom Level Selector — Paper Configuration UI
Type: Segmented control with 6 options
Options: Remember (L1, red), Understand (L2, orange), Apply (L3, amber), Analyse (L4, green), Evaluate (L5, blue), Create (L6, purple)
On select: Show preview of the Bloom template text below the selector so teacher knows what the question style will be
Tooltip: Hover/tap shows the full prompt template for that level
Keyboard: Arrow keys navigate between options
Blink Challenge Widget — Liveness UI
Layout: Webcam feed in center (480×360 for desktop, full-width for mobile) with MediaPipe face mesh overlay
Instruction text: Large, centered: "Blink twice to begin" with animated eye icon
Blink counter: Shows ●○ → ●● (filled dots as blinks detected)
Timer: Countdown from 8 seconds. Orange at 3 seconds. Red at 1 second.
On success: Green flash overlay → smooth transition to first exam question
On timeout: "Retry" button + "Need help?" link that shows teacher contact info
Offline Resilience Banner — Student Exam View
Trigger: navigator.onLine === false OR fetch error on answer submit
Appearance: Amber banner at top of exam view: "⚠ No internet connection. Your answers are being saved locally every 30 seconds."
Background behavior: localStorage save every 30s. Show timestamp: "Last saved: 14 seconds ago"
On reconnect: Banner changes to blue: "✓ Reconnected. Syncing your answers..." → green: "✓ All answers synced."
At 5min disconnect: Banner: "Session paused — your teacher has been notified. You can continue when you reconnect."
